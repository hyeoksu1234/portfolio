상명대학교 졸업전시

Project Lead · Product Planning & Delivery

서로 다른 스튜디오와 형식으로 제출된 97명의 데이터를 194개 작품 항목으로 연결하고, 행사 기간에도 안정적으로 운영할 수 있는 데이터·이미지 파이프라인을 만들었습니다.

Live Site ↗ GitHub ↗

Contribution

팀 리드, 정보 구조, 인터랙션 디자인, 데이터 모델, 프론트엔드, 이미지 배포 자동화

Problem

작품은 많았지만 관람자가 탐색할 공통 구조가 없었습니다.

97명의 졸업생이 두 스튜디오에 걸쳐 서로 다른 형식으로 데이터를 제출했습니다. 같은 학생의 복수 소속과 대용량 이미지, 행사 직전 수정 요청을 동시에 처리해야 했습니다.

  • 학번 기준 데이터 중복과 복수 스튜디오 소속을 함께 처리해야 했습니다.
  • 외부 데이터 장애 시에도 전시 페이지가 열려야 했습니다.
  • 프로필 이미지 업로드·URL 생성의 반복 작업을 줄여야 했습니다.

Decision

  • Supabase-first + local fallback 운영 데이터의 최신성을 유지하면서도 외부 장애가 전시 전체 중단으로 이어지지 않게 했습니다.
  • 학번 중심 데이터 머지 복수 스튜디오 소속을 잃지 않으면서 동일 학생의 프로필과 작품 정보를 일관되게 묶기 위해서입니다.
  • S3 업로드와 CDN URL 생성 자동화 마감 직전의 반복 업로드 실수와 수작업 시간을 줄이기 위해서입니다.

Process

  1. 01
    콘텐츠 모델링

    학생, 스튜디오, 작품, 프로필의 관계와 예외 케이스를 먼저 정의했습니다.

  2. 02
    탐색 경험 설계

    모바일·데스크톱에서 작품과 작가를 오갈 수 있는 정보 구조와 인터랙션을 만들었습니다.

  3. 03
    데이터·이미지 파이프라인

    Supabase, local fallback, S3/CloudFront 자동화와 request-level cache를 적용했습니다.

  4. 04
    전시 운영

    행사 전 콘텐츠 병합과 반응형 QA를 거쳐 공개 배포했습니다.
